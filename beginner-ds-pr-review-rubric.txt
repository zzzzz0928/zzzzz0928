D2RS-2026spring Beginner Data Science PR Review Rubric
=======================================================

This rubric is used by the automated PR review agent to provide structured,
teaching-focused feedback to beginner data-science students.

----------------------------------------------------------------------
SECTION 1: CODE CORRECTNESS & FUNCTIONALITY
----------------------------------------------------------------------
Must Fix:
- Code must run without syntax errors (R, Python, Quarto).
- All referenced variables, functions, and packages must be defined or imported.
- File paths must be relative (not absolute) so the code works on any machine.
- Data loading must reference files that exist in the repository.

Should Improve:
- Avoid hard-coded constants that should be parameters or named variables.
- Functions should return values explicitly rather than relying on side effects.
- Avoid modifying global state unnecessarily.

Nice to Have:
- Add brief inline comments explaining non-obvious logic.
- Use early returns / guard clauses to reduce nesting.

----------------------------------------------------------------------
SECTION 2: REPRODUCIBILITY
----------------------------------------------------------------------
Must Fix:
- set.seed() or random_state must be set whenever random operations are used.
- External data must be read from a committed file or a clearly documented URL.
- All required packages/libraries must be listed (e.g., library() calls in R,
  import statements in Python).

Should Improve:
- Pin package versions or note the R/Python version used.
- Avoid session-state dependencies (each script should run top-to-bottom from
  a clean environment).

Nice to Have:
- Include a small sample of raw data or describe its shape in comments.
- Document data provenance (where data came from, when it was collected).

----------------------------------------------------------------------
SECTION 3: CODE STYLE & READABILITY
----------------------------------------------------------------------
Must Fix:
- Files must be saved in UTF-8 encoding.
- Variable names must be in English or pinyin that clearly conveys meaning
  (single-letter names like x, df1, tmp are acceptable only in short scopes).

Should Improve:
- Follow consistent naming conventions:
  - R: snake_case for variables and functions (e.g., student_data, calc_mean).
  - Python: snake_case for variables/functions, UPPER_CASE for constants.
- Lines should generally not exceed 100 characters.
- Group related code into clearly separated blocks; blank lines aid readability.

Nice to Have:
- Add a docstring or header comment at the top of each script describing its
  purpose, inputs, and outputs.
- Use a linter (lintr for R, ruff for Python) to catch style issues.

----------------------------------------------------------------------
SECTION 4: DATA ANALYSIS QUALITY
----------------------------------------------------------------------
Must Fix:
- Visualisations must have axis labels, a title, and units where applicable.
- Summary statistics or tables must have clear column headers.
- Do not present results based on an incorrect or misunderstood question.

Should Improve:
- Acknowledge and handle missing values explicitly (na.rm = TRUE, dropna(), etc.).
- Describe what each plot or table is showing in accompanying text.
- Avoid misleading chart types (e.g., do not use pie charts for > 5 categories).

Nice to Have:
- Include a brief interpretation sentence after each key result.
- Compare results against a baseline or expectation.
- Test edge cases: what happens with an empty dataset?

----------------------------------------------------------------------
SECTION 5: DOCUMENTATION & COMMUNICATION
----------------------------------------------------------------------
Must Fix:
- README or QMD narrative must accurately describe what the analysis does.
- Variable or column names used in text must match those in the code.

Should Improve:
- Use Markdown headings and lists to structure written explanations.
- Explain the "why" behind analytical choices, not just the "what".
- Proofread for typos and grammatical errors.

Nice to Have:
- Provide a TL;DR summary at the top of longer documents.
- Use cross-references between sections for easier navigation.

----------------------------------------------------------------------
SECTION 6: REPOSITORY HYGIENE
----------------------------------------------------------------------
Must Fix:
- Do not commit large binary files, model weights, or files > 10 MB.
- Do not commit credentials, API keys, passwords, or personal tokens.
- Do not commit generated/rendered output (e.g., _site/, __pycache__/,
  *.Rout) unless it is the intended deliverable.

Should Improve:
- Add or update .gitignore to exclude temporary files, IDE config, and build
  artifacts (e.g., .DS_Store, .Rhistory, *.pyc, _site/).
- Keep commit messages short but descriptive (< 72 characters, imperative mood).

Nice to Have:
- Atomic commits: one logical change per commit.
- Tag or reference the issue/task that the PR addresses.

----------------------------------------------------------------------
SECTION 7: BEGINNER BEST PRACTICES
----------------------------------------------------------------------
Must Fix:
- Do not use install.packages() inside a script that will be run automatically
  (it should be in setup/README only).
- Do not use rm(list = ls()) at the top of scripts — it breaks other people's
  environments.

Should Improve:
- Prefer vectorised operations over explicit loops where natural in R/Python.
- Prefer tidyverse / pandas idioms over base-R loops or raw indexing for
  readability.
- Break long analysis pipelines into small, named intermediate objects for
  easier debugging.

Nice to Have:
- Write at least one simple assertion or sanity check (e.g., stopifnot(),
  assert) after critical data transformations.
- Consider edge cases: what if a student's entry is missing or malformed?

----------------------------------------------------------------------
END OF RUBRIC
----------------------------------------------------------------------
